# 软件需求规格说明书 (SRS) - 校园互助服务平台

## 1. 引言 (Introduction)

### 1.1 目的 (Purpose)

本文档旨在明确“校园互助服务平台”的软件需求，作为系统设计、开发、测试以及验收的基础依据。本文档的预期读者包括项目经理、软件开发工程师、UI/UX设计师、软件测试工程师及系统管理员。

### 1.2 文档约定 (Document Conventions)

- **发布者 (发单人)**：在平台上发布互助需求、二手商品或组队邀请的用户。
- **接单者 (服务方)**：在平台上接受互助需求并提供服务/商品的用户。
- 本文档中所有功能优先级分为：高（核心功能，必须实现）、中（重要扩展）、低（优化项）。

### 1.3 产品范围 (Product Scope)

“校园互助服务平台”是一款专为在校大学生打造的C2C（Client to Client）综合性服务软件。系统旨在打破校园内的信息壁垒，提供一个安全、便捷、高效的互助环境。核心模块包括：**快递代取、学习辅导、二手交易、组队匹配**。

### 1.4 参考文献 (References)

- IEEE Std 830-1998, IEEE Recommended Practice for Software Requirements Specifications

## 2. 综合描述 (Overall Description)

### 2.1 产品前景 (Product Perspective)

本系统计划作为一个独立的移动端应用（优先微信小程序/APP）运行，后端依托云服务器部署。系统将整合校园内的闲置时间与资源，形成校园内部的小型服务经济生态圈。

### 2.2 产品功能 (Product Functions)

- **用户管理**：学号/校园统一身份认证、个人主页维护。
- **任务大厅**：互助需求的分类展示、搜索、筛选与接单。
- **发布系统**：支持多类型的需求和资源发布（带悬赏/标价）。
- **消息系统**：发单者与接单者之间的即时通讯（IM）及系统状态通知。
- **交易与评价**：平台虚拟货币/第三方支付托管，以及双向信用评价体系。

### 2.3 用户类和特征 (User Classes and Characteristics)

- **学生用户（主力）**：具备基本智能手机操作能力，对时间敏感，需求多样化。要求系统操作简便、响应迅速。必须通过校园学生身份认证（学号/Edu邮箱）以保证社区纯净度。
- **平台管理员**：负责审核异常信息、处理交易纠纷、管理公告和用户账号封禁等。

### 2.4 运行环境 (Operating Environment)

- **客户端**：微信小程序环境 或 移动端（iOS 11.0+, Android 8.0+）。
- **服务器端**：Linux (Ubuntu/CentOS), Node.js/Java Spring Boot, MySQL/PostgreSQL, Redis。

------

## 3. 外部接口需求 (External Interface Requirements)

### 3.1 用户界面 (User Interfaces)

- 采用符合大学生审美的年轻化、扁平化设计。
- 底部导航栏分为：首页（大厅）、发布（+号）、消息、我的。
- 要求在不同分辨率的手机屏幕上具备良好的自适应能力。

### 3.2 软件接口 (Software Interfaces)

- **身份认证接口**：对接学校的教务系统API或校园一卡通认证网关（若无权限，则采用Edu邮箱验证机制）。
- **支付接口**：对接微信支付/支付宝开放平台API。
- **地图/定位接口**：对接高德或腾讯地图API，用于计算宿舍/教学楼之间的距离（非必需，但推荐）。

## 4. 系统特性 / 功能需求 (System Features)

### 4.1 校园认证与账户管理

- **4.1.1 描述**：确保所有用户均为本校真实学生。
- **4.1.2 刺激/响应序列**：
  - 用户输入学号及教务系统密码 -> 后端请求学校接口校验 -> 校验成功，创建账号并标记为“已认证”。
- **4.1.3 功能要求**：不支持未认证用户发布或接受任何任务，未认证用户仅有有限的浏览权限。

### 4.2 互助任务发布系统

- **4.2.1 描述**：用户发布不同类型的需求。
- **4.2.2 功能要求**：
  - **快递代取**：输入取件码、大小件标识、送达地点、悬赏金额、期望送达时间。
  - **学习辅导**：选择科目（如高数、编程）、输入辅导需求说明、选择线上/线下、设定薪酬。
  - **二手交易**：上传实物图片、输入新旧程度、价格、交易地点（支持议价/一口价）。
  - **组队匹配**：选择类别（比赛、项目、拼车、旅游）、填写队伍要求（如缺前端开发）、截止时间。

### 4.3 任务大厅与接单机制

- **4.3.1 描述**：信息流展示与撮合交易。
- **4.3.2 功能要求**：
  - 提供按分类、按最新发布、按最高金额排序的功能。
  - 接单者点击“立即接单”，系统需校验接单者是否满足前置条件（如信誉分过低不可接单）。
  - 接单后，订单状态由“待接单”转为“进行中”，大厅隐藏该需求。

### 4.4 订单流转与支付托管

- **4.4.1 描述**：保障资金安全的交易闭环。
- **4.4.2 功能要求**：
  - 发单时，悬赏金额需预先支付并托管至平台。
  - 接单者完成任务后点击“已完成”，发单者确认收货/服务后，资金自动打入接单者账户。
  - 若发生纠纷，提供“申请客服介入”按钮，冻结该笔资金直至处理完毕。

### 4.5 评价与信誉积分体系

- **4.5.1 描述**：维护校园互助的信任基础。
- **4.5.2 功能要求**：订单完成后，双方可进行1-5星打分及文字评价。初始信誉分为100分，低于60分将限制接单和发布权限。

## 5. 其他非功能需求 (Other Nonfunctional Requirements)

### 5.1 性能需求 (Performance Requirements)

- **并发处理**：系统应能支持高峰期（如中午下课后、晚自习后）至少 1000 QPS 的并发访问量。
- **响应时间**：核心接口（如刷新大厅、抢单）响应时间不得超过 1.5 秒。
- **图片加载**：二手交易模块的高清图片应使用 CDN 加速，并生成缩略图，确保弱网环境下列表加载顺畅。

### 5.2 安全性需求 (Security Requirements)

- **数据加密**：用户密码必须使用 bcrypt 或类似强哈希算法加密存储。敏感信息（如真实姓名、取件码）在传输过程中需强制使用 HTTPS。
- **隐私保护**：系统内置虚拟电话/IM私聊功能，未经双方同意，不得向对方直接展示真实手机号码。
- **防刷单机制**：同一IP或同一设备在短时间内频繁发布/接单，系统应自动拦截并触发人机验证或人工审核。

### 5.3 软件质量属性 (Software Quality Attributes)

- **可用性 (Availability)**：系统需要达到 99.9% 的正常运行时间，服务器具备自动容灾和定期备份（每日一次数据库快照）能力。
- **可维护性 (Maintainability)**：后端代码应遵循 RESTful API 规范，保留清晰的 Swagger 或 Postman 接口文档，便于后续功能的迭代与交接。

------

## 6. 其他需求 (Other Requirements)

### 6.1 数据合规性

必须在用户首次登录时展示《用户协议》与《隐私政策》，明确告知用户信息（特别是学号、位置数据）的使用范围仅限于本校互助服务，符合现行数据保护法律法规。
