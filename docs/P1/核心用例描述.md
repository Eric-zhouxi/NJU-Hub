# 3.1 核心用例详细描述 

**【用例 1】UC01: 注册与校园认证**

- **角色**：所有学生用户
- **前置条件**：用户已下载 App 或进入小程序，且未登录。
- **基本流**：
  1. 用户选择“注册/认证”。
  2. 系统提示用户阅读并同意《隐私政策》与《用户协议》。
  3. 用户输入学校教务系统账号（学号）与密码，或使用校园 Edu 邮箱获取验证码。
  4. 系统后端调用对应接口验证身份真实性。
  5. 验证通过后，系统创建用户档案，初始化信誉分为 100 分。
  6. 系统提示认证成功，并跳转至首页大厅。
- **替代流**：
  - 3a. 若教务系统接口超时/无响应：系统提示“认证服务繁忙，请尝试邮箱认证”，引导用户走备选方案。
  - 4a. 若账号密码错误或邮箱非本校后缀：系统拒绝认证，提示“校园身份校验失败，请检查输入”。
- **后置条件**：用户获得平台完整使用权限。

**【用例 2】UC08 & UC09: 接单与上传凭证结单**

- **角色**：接单学生
- **前置条件**：接单学生已登录，信誉分不低于 60分，且大厅有处于“待接单”状态的跑腿任务。
- **基本流**：
  1. 接单学生在任务大厅点击某个跑腿任务（如代取快递）的“立即接单”。
  2. 系统将该任务状态更新为“进行中”，锁定发单方托管的酬劳，并向发单方推送通知。
  3. 接单学生完成线下跑腿任务，将物品送达指定地点。
  4. 接单学生在订单详情页点击“我已完成”，并强制调用相机拍摄送达现场照片上传作为凭证。
  5. 系统更新订单状态为“待确认”，等待发单方验收。
- **替代流**：
  - 1a. 若接单学生信誉分低于 60 分：系统弹窗拦截，提示“信誉分过低，暂无接单权限”。
  - 4a. 若未上传照片凭证直接点击完成：系统阻断操作，提示“必须上传现场照片作为完成凭证以保障双方权益”。
- **后置条件**：任务进入待验收状态，双方均保留了凭证记录。

**【用例 3】UC14: 仲裁交易纠纷 (学生评审团机制)**

- **角色**：志愿管理员（或高信用等级的学生评审员）
- **前置条件**：某笔订单发单方与接单方产生纠纷，且其中一方已点击“申请客服/评审介入”。资金处于冻结状态。
- **基本流**：
  1. 系统自动将该纠纷工单推送到“仲裁中心”。
  2. 志愿管理员接单，查阅双方聊天记录、跑腿要求及接单方上传的现场照片凭证。
  3. 志愿管理员通过系统判定责任方（例如：照片显示物品完好送达，责任在发单方未及时取走）。
  4. 志愿管理员做出裁决（将托管资金打给接单方，或退还给发单方）。
  5. 系统执行资金操作，并扣除责任方信誉分，工单结束。
- **替代流**：
  - 3a. 若凭证不足以判定（双方各执一词）：志愿管理员可将判定权升级为“大众评审”，匿名公示关键信息，由 5 名随机高信誉用户投票决定资金归属。
- **后置条件**：纠纷解决，资金解冻并完成划拨，违规方受到惩罚。



基于本次校园互助服务平台的开发过程，以下是针对 **3.3 AI 辅助实验** 的对比分析与评价。

------

### 1. 用例与用户故事对比：AI vs. 人工调研

通过将初始 AI 生成的通用需求与基于《调研原始数据》整理的人工版本进行对比，发现两者在思维逻辑上存在显著差异：

- **人工版本的独特性**：人工版本识别出了具有校园特色的“信息不对称”需求，如**“教师/课程红黑榜”**和**“失物招领”**，这些在初始 AI 生成中完全缺位。
- **结算机制的差异**：AI 默认采用商业化的“资金托管”，而人工调研发现学生更倾向于**“积分酬劳”**，且对**“拍照留痕”**作为结单依据有极高的要求。
- **管理模式的下沉**：AI 倾向于“全职客服仲裁”，而人工版本基于访谈 A 的建议提出了**“学生评审团/大众评审”**机制，更符合校园自治生态。

------

### 2. 重点问题回答

#### Q1：AI 生成的内容是否遗漏了异常流？

**回答：是。AI 往往只关注“快乐路径”（Happy Path），对校园环境下的特有异常流缺乏预见。**

- **AI 遗漏的举例**：在“快递代取”用例中，AI 仅考虑了“送达”与“未送达”。
- **人工识别的异常流**：
  - **取件码错误/失效**：接单人到达驿站发现取件码无效时的处理流程。
  - **隐私泄露风险**：用户因担心学号注册泄露信息而产生的防御性行为。
  - **恶意刷单/水贴异常**：调研中 51 份问卷高频提到的“言论管理”与“刷单”封禁逻辑。

#### Q2：验收标准是否具体可测试？

**回答：AI 生成的初始标准较为模糊，经人工修改后的版本才达到“具体可测试”的要求。**

以下通过用户故事的验收标准（Acceptance Criteria, AC）进行对比举例：

| **维度**     | **AI 初始生成的 AC (示例)** | **人工修改后可测试的 AC (示例)**                             |
| ------------ | --------------------------- | ------------------------------------------------------------ |
| **具体性**   | “系统应保证交易安全”        | “发布需求时，系统需实时校验账户积分余额是否 $\ge$ 悬赏金额”  |
| **可观测性** | “用户可以上传图片”          | “接单人点击‘已完成’时，系统必须强制调用相机权限上传至少 1 张现场照片” |
| **测试判定** | “支持多种认证方式”          | “输入非 @xxx.nju.edu.cn 后缀的邮箱时，系统应提示‘非校内邮箱格式’并阻止发送验证码” |

------

### 3. 实验总结

- **AI 的价值**：提供标准的 IEEE 830 框架、逻辑通顺的基础流（Basic Flow）以及通用的非功能需求建议。
- **人工的不可替代性**：
  - **捕捉痛点**：如“教师红黑榜”这种带有特定校园文化的需求，AI 无法通过通用语料库预判。
  - **落地闭环**：基于访谈得到的“拍照存证”和“大众评审”是解决校园信任危机、使验收标准具体化、可测试的关键依据。
