# NJU-Hub校园互助服务平台需求规格说明书 (SRS)

## 1. 引言 

### 1.1 目的 

本文档旨在定义NJU-Hub校园互助服务平台的功能与非功能需求，为开发团队提供指导，并作为验收依据。

### 1.2 范围 

>  **说明**：在原有交易功能基础上，根据调研结果显著扩大了信息咨询与学习资源共享的范围。

本平台是一个专为在校大学生设计的综合互助社区。它不仅支持传统的跑腿、交易、组队，还深度集成了校园信息咨询、学术资料共享以及学生自治的信用评价体系。

### 1.3 定义与缩写 

- **Dazi (搭子)**：指为特定活动（如拼车、运动）寻找同伴的社交行为。
- **积分 (Points)**：平台内部流通的虚拟酬劳，可替代现金进行互助激励。

### 1.4 参考文献 

1. 《调研原始数据.md》

## 2. 总体描述 

### 2.1 产品视角

> **说明**：根据访谈反馈，将定位从“独立APP”修正为“轻量化平台（如小程序）”以降低使用门槛。

系统应作为微信小程序或轻量化 Web 应用运行，优先保证界面简洁、无广告，并能与校园网环境良好适配。

### 2.2 用户特征 

- **求助/发布者**：有时间压力或资源需求的在校生。
- **服务/接单者**：希望利用余力赚取积分或报酬的在校生。
- **志愿管理员**：由学生组成的审核团队，负责纠纷仲裁和氛围维护。

### 2.3 约束与假设 

- **约束**：必须保障学号等隐私数据的加密存储，防止数据库泄露。
- **假设**：学校教务系统或邮箱系统提供基本的身份验证 API 接口。

## 4. 非功能需求

### 4.1 性能需求 [AI生成+人工修改]

- **并发响应**：在中午/下课高峰期（跑腿需求爆发期），系统需支持高并发刷新。

### 4.2 安全性需求 [人工补充]

- **隐私隔离**：在线沟通时默认隐藏真实手机号。
- **数据库加密**：对学号及教务认证信息进行高强度加密，防止撞库。

### 4.3 可用性需求 [人工补充]

- **界面极简**：遵循调研建议，操作流程不应多于 3 次点击，且必须无广告骚扰。
